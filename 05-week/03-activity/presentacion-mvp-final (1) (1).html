El Problema
YS — 0:00 a 2:00 min

¿Qué pasa hoy
sin nuestro sistema?

🏥
Escenario real — Consultorio médico
Un consultorio recibe llamadas todo el día para agendar citas.
La recepcionista anota en papel o cuaderno.
El paciente llama… y nadie contesta.
Se generan citas duplicadas en el mismo horario.
El paciente no sabe si quedó agendado.
No hay historial de ningún paciente.
📞
Sin sistema
Citas por llamadas y mensajes sin registro ni control alguno.
📋
Proceso manual
Agendas en papel con errores y duplicación constante.
⏳
Esperas y pérdidas
Turnos perdidos, clientes frustrados, cero trazabilidad.
📉
No escala
Imposible crecer con un sistema tan desorganizado.

⚡ Decisiones de arquitectura
🔵
API Gateway como único punto de entrada
El cliente nunca llama directo al microservicio. El Gateway centraliza auth, logging y routing.
🔗
Microservicios desacoplados
Cada servicio tiene su propio proceso. Comparten una única BD PostgreSQL centralizada.
🐳
Docker Compose para orquestación local
Un solo comando levanta todo: Gateway + Turnos Service + PostgreSQL.
🧱
Arquitectura en capas dentro del servicio
Controller → Service → Repository → BD. Separación clara de responsabilidades.

05
Backlog del Sprint
YS — 6:00 a 8:00 min

¿Qué hicimos,
qué faltó y qué sigue?

✅
Qué se hizo
Se identificó el problema de desorganización en la gestión de citas.
Se diseñó una solución digital para organizar y gestionar turnos.
Se desarrolló una plataforma donde los usuarios pueden solicitar y consultar sus citas.
Se implementó un sistema básico para registrar y visualizar los turnos solicitados.
⚠️
Qué faltó
Implementar un módulo más completo de administración de usuarios y citas.
Agregar más opciones para modificar o cancelar citas.
Mejorar algunos aspectos de organización y visualización de la información.
🚀
Qué sigue
Ampliar las funciones de gestión de citas para mayor control y organización.
Incorporar mejoras que faciliten la experiencia del usuario.
Desarrollar nuevas opciones que permitan una administración más completa del sistema.
