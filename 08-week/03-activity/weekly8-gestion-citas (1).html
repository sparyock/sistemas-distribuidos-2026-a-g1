Decisiones arquitecturales

ADRs Implementados

ADR-006

ResponseEntity para respuestas HTTP

Control total sobre códigos de estado y el cuerpo de cada respuesta del API.

return ResponseEntity .status(HttpStatus.CREATED) .body(turno);
✓ Respuestas claras: 201 Created, 200 OK, 404 Not Found según cada caso.
ADR-008

@RequestParam para filtros en endpoints

Consultas con parámetros de URL sin necesidad de crear múltiples endpoints.

GET /turnos?idUsuario=2 @GetMapping public ResponseEntity<List<Turno>> obtenerTurnos(@RequestParam Long idUsuario)
✓ Sistema más flexible, reutilizable y fácil de escalar.
ADR-009

Validación con @Valid y anotaciones

Validación automática de datos antes de persistir: sin campos vacíos, sin fechas pasadas, sin nulos.

@NotNull(message = "idUsuario obligatorio") private Long idUsuario; @Future(message = "Fecha debe ser futura") private LocalDateTime fechaHora;
✓ Mejora seguridad e integridad de los datos del sistema.
ADR-010

GlobalExceptionHandler para errores

Manejo centralizado de errores: respuestas limpias y legibles en lugar de stack traces técnicos.

// Antes: mucho texto técnico innecesario // Ahora: { "idUsuario": "Campo obligatorio", "fechaHora": "Debe ser futura" }
✓ API más profesional y fácil de mantener.
Weekly 8 · Próximos pasos

¿Qué se hará?

La siguiente fase conecta frontend y backend para que las notificaciones sean reales, no simuladas.

1

Desarrollar el backend del microservicio

Implementar Spring Boot con JPA, H2/MySQL, JavaMailSender y el scheduler de recordatorios.

2

Definir contrato JSON frontend ↔ backend

Acordar la estructura de los DTOs para que el servicio Angular consuma los endpoints reales.

3

Conectar frontend con backend real

Reemplazar los datos mock del servicio Angular por llamadas HTTP a los endpoints REST.

4

Probar flujo completo end-to-end

Validar los 4 eventos: turno confirmado, cancelado, reprogramado y recordatorio — con email real y notificación in-app.
